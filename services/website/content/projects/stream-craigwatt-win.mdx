stream.craigwatt.win is my self-hosted video streaming platform, but I treat it like a small production deployment rather than a toy lab.

The goal is to keep it repeatable, observable, and resilient on commodity hardware while still giving me a clean place to experiment with Kubernetes and automation.

## Platform Pieces

- **Raspberry Pi k3s cluster** — multi-node Kubernetes as the compute foundation
- **Jellyfin on Kubernetes** — containerised media server with persistent storage
- **MetalLB** — stable, LAN-friendly load balancing for predictable local access
- **Ingress / routing** — Traefik-based ingress so services are exposed consistently
- **Persistent storage layer** — media libraries and config are treated as first-class state
- **Automation-first** — Ansible playbooks and roles to bootstrap nodes and apply config
- **Operational guard rails** — upgrades and deploys are designed to avoid foot-guns

## Engineering Highlights

- **Reproducible cluster build** — new nodes can be provisioned consistently via automation rather than manual steps
- **Clean separation of concerns** — cluster bootstrap, platform services, and app workloads are managed independently
- **Networking that behaves** — MetalLB avoids “random NodePort” mess and makes LAN access predictable
- **Storage decisions made deliberately** — media libraries and config are handled as first-class persistent concerns

## Why It Matters

This project is a realistic demonstration of how I build and run systems:

- infrastructure as code instead of hand-crafted servers
- Kubernetes fundamentals applied pragmatically
- operational reliability and upgradeability as design goals
- a home-lab that behaves like a small production environment
